迪米特法则

迪米特法则：高内聚、低耦合！

迪米特法则 Law Of Demter 也称为最少知识原则 Least Knowledge Principle : 一个对象应该对其它对象有最少的了解！

迪米特法则的核心观念就是 类间解耦、弱耦合；只有弱耦合之后，类的复用率才可以提高！其要求的结果就是产生了大量的中转类或者跳转类，导致系统的复杂性提高，同时也为维护带来了难度！在实际开发中，使用迪米特法则需要反复权衡，既做到让结构清晰，又做到 高内聚低耦合！

迪米特法则对类的低耦合，包含四层含义：

  • 只和朋友交流
  • 朋友之间要保持距离
  • 是自己的，就是自己的
  • 谨慎使用 Serializable

1、只和朋友交流

迪米特法则只与直接的朋友通信！什么是直接的朋友：

  • 每个对象都必然会与其它对象保持耦合关系；
  • 两个对象之间的耦合关系就成为朋友关系；

这种关系的类型很多，如：组合、聚合、依赖等！观察下述代码：

  • 班长有一个重要职责：清点班级实到人数；
  • 老师上课前，需要询问班长：你们班的实到人数
/// 学生
class Student {
    var leave : Bool /// 是否缺勤
}

/// 班长
class GroupLeader {

    /// 清点实到人数
    func actualCounts(student: [Student]) -> Int {
        func isLeave(stu:Student) -> Bool {
            return stu.leave
        }
        return student.filter(isLeave).count
    }
}

class Teacher {

    /// 通过班长，了解班级实到人数
    func request(leader:GroupLeader) {
        var students : [Student] = []
        while students.count < 30 {
            let stu = Student()
            stu.leave = (arc4random() % 2 == 0)
            students.append(stu)
        }
        let actualCounts = leader.actualCounts(student: students)
        print("实到人数 \(actualCounts)")
    }
}

func LoD_Scene_Test() {
    let teacher = Teacher()
    teacher.command(leader: GroupLeader())
}

思考上述代码，老师有几个直接耦合对象？只有班长一人，Teacher 类不需要学生产生直接关系！但是学生类出现在了老师的方法 request() 中！

在迪米特法则中，朋友类的定义：出现在成员变量、方法的输入输出参数中的类是成员朋友类；而出现在方法内部的类不属于朋友类！ 方法是一个类的行为，类 Teacher 竟然不知道自己的行为 request() 与其它类 Student 产生依赖关系，这是不允许的，严重违反了迪米特法则！

类与类之间的关系是建立在类间的，而不是方法间；原则上一个方法不应该引入该类中不存在的对象！

问题已经发现，我们对上述代码做一个简单修改：

class Student {
    var leave : Bool /// 是否缺勤
}

class Lod_GroupLeader {
    var student: [Student] /// 班长时刻持有自己的班级学生名单
    init(student: [Student]) {
        self.student = student
    }

    func actualCounts() -> Int {
        func isLeave(stu:Student) -> Bool {
            return stu.leave
        }
        return student.filter(isLeave).count //查询自己班级实到人数
    }
}

class LoD_Teacher {
    func request(leader:Lod_GroupLeader) {
        let actualCounts = leader.actualCounts()
        print("实到人数 \(actualCounts)")
    }
}

/// 场景测试
func LoD_Scene_Test() {

    var students : [Student] = []
    while students.count < 30 {
        let stu = Student()
        stu.leave = (arc4random() % 2 == 0)
        students.append(stu)
    }

    // 选出班长的时刻，班长需要掌握自己的班级同学
    let leader = Lod_GroupLeader(student: students)
    let teacher = LoD_Teacher()

    // 老师只需询问某个班长人数即可
    teacher.request(leader:leader)
}

经过修改之后，老师与学生再无耦合关系！在迪米特法则中，一个类尽量只和朋友类交流，不与陌生类交流！

2、朋友之间要保持距离

人与人之间是有距离的，关系太远则逐渐疏远、最终形同陌路；太近则相互伤害！迪米特法则对这个距离进行描述：朋友之间既不能无话可说，也不能无所不知！

一个类公开的 public 属性或者方法越多，修改时涉及的面也就越大，变更引发的风险也就越大！为了保持朋友间的距离，在设计时需要反复衡量是否可以减少公共属性、方法，是否可以加上 fina 等关键字；做到类的高内聚！

如下述代码：

  • 人们周末都要休息，但具体做什么，我们无权去过分干涉
  • 我们只需要询问一下即可，不要具体指点朋友的某个具体行为
protocol Lod_People {
    func rest()
    func askPeople(people:Lod_People)
}

extension Lod_People {
    func askPeople(people:Lod_People)  {
        people.rest()
    }
}

class Lod_People_1: Lod_People {
    private func sleep_weekend() {
        print("睡到中午")
    }
    private func eat(){
        print("通宵撸个串")
    }

    /// 仅仅对外暴露休息的行为
    /// 但具体做什么，就没必要一一道来了
    func rest() { 
        eat()
        sleep_weekend()
    }
}

class Lod_People_2: Lod_People {

    private func study() {
        print("好好学习")
    }
    func rest() {
        study()
    }
}

func LoD_Scene_Test() {    
    let p1 = Lod_People_1()
    let p2 = Lod_People_2()
    p2.askPeople(people: p1)
    p1.askPeople(people: p2)
}

迪米特法则要求类 羞涩 一点，尽量不要对外公布太多的 public 方法和非静态的 public 变量，尽量内敛！多使用 private \ protected 等访问权限！

3、是自己的，就是自己的

在现实开发中，经常碰到这样一个问题“一个方法放在本类中可以，放在其它类中也没有错，那么放在哪里合适呢”？

如果一个方法放在本类中，既不增加类间关系，也不对本类产生负面影响，那就放置在本类中！
