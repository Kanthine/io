开闭原则

Software entities like classes, modules and functions should be open for extension but cloesd for modifications !

1、什么是开闭原则

一个项目只要在生命周期内都会发生变化，既然变化是无法避免的事实，在项目设计之初就应该尽量适应后期的变化，以提高项目的稳定性和灵活性，真正实现“拥抱变化”！

开闭原则要求：一个软件实体应该通过扩展来实现变化，而不是通过修改已有代码来实现变化（ 软件实体包含：项目按一定的逻辑规则划分的模块、抽象和类、方法 ）。开闭原则是为项目的未来变化而对现在开发的约束原则！

1.1、case: 图书管理系统的历史设计

图书馆的书籍管理系统：

  • 有基本的书籍信息：作者、书名、价格...
  • 有书籍分类：计算机类、数学类、法律类...
  • 还有书籍的持有层：数据库表
  • 高层模块：借阅模块、查询模块...

首先定义书籍信息接口，包含作者、书名、价格等基本信息：

/// 抽象书籍的基本属性
protocol IBaseBook {
    func getName() -> String
    func getPrice() -> Float
    func getAuthor() -> String
}

接着定义一个具体实现类，目前图书馆成立时间短，仅收藏了小说类书籍：

class BaseBook: IBaseBook {
    private let name : String
    private let author : String
    private let price : Float

    init(name:String, author:String, price:Float) {
        self.name = name
        self.price = price
        self.author = author
    }

    func getName() -> String {
        return name
    }

    func getPrice() -> Float {
        return price
    }

    func getAuthor() -> String {
        return author
    }
}

对于书籍的管理，有一个持久层的数据库：

/// 持久层：书库
class BookStorage  {
    static func getBookList() -> [BaseBook] {
        var list : [BaseBook] = Array()
        for index in 1..<100 {
            let book = BaseBook(name: "name_\(index)" , author: "author_\(index)", price: Float(arc4random() % 999))
            list.append(book)
        }
        return list
    }
}

对于用户而言，直接在接图书馆查阅即可：

/// 业务层：图书馆
class Library {
    private let bookList = BookStorage.getBookList()
    func lookBooksInfo() {
        for book in bookList {
            print("\(book.getAuthor()) 撰写的 \(book.getName()) 售价为 \(book.getPrice())")
        }
    }
}

/// 场景测试：查询书籍列表
func OCP_Scene_Test() {
    let xinhua = Library()
    xinhua.lookBooksInfo()
}

目前为止，一个简单的查询功能实现了！现在来考虑未来的变化：

  • 假如图书馆扩大收藏范围，以后收纳了历史、计算机书籍，如何实现该变化？
  • 假如图书馆未来职能扩充，不仅提供查询功能，还有借阅模块、销售模块，如何实现该变化？
  • 假如碰到双十一、618 等电商促销节，图书馆要打折促销一部分书籍，如何实现该变化？
  • 未来的变化不胜枚举，我们需要尽可能的设计良好的代码架构；

1.2、基于历史，面对变化

未来的变化不可尽估，所以要使用开闭原则约束当下的设计规则！比如如何实现打折促销活动？基于上述已经投产的项目，有三种方案可以实现：

case1、修改接口

在接口 Book 上新增加一个方法 getPromotionPrice() 专门用于促销活动：

  • 由此导致所有的实现类需要添加该接口方法，高层业务 Library 的查阅方法也要改；
  • 一个接口应该是稳定的、可靠的，不应该经常发生变化，否则接口作为契约的作用就失去了效能；

因此，毙掉该方案。

case2、修该实现类

直接在实现类 NovelBook 中的 getPrice() 修改价格：

  • 简单省事，在具有明确开发章程的团队内部是一个优秀的解决方案；
  • 但会计部门将无法查阅该书籍的真实价格，信息的不对称导致上层领导的决策失误；

因此，该方案也不是一个最优方案！

case3、通过扩展实现变化

为 BaseBook 增加一个子类 NovelBook_Promotion ，重写 getPrice() 方法，数据库通过生产 NovelBook_Promotion 产生一批新的数据，完成业务变化对系统的最小化开发！

通过扩展来实现变化，而不是通过修改已有的代码来实现变化；好处是对历史代码的修改小，由此引发的风险降到了最低！

/// 通过扩展实现变化
class NovelBook_Promotion: BaseBook {
    override func getPrice() -> Float {
        return super.getPrice() * 0.6
    }
}

/// 持久层：书库
class BookStorage  {
    /// 增加促销信息
    static func getPromotionBookList() -> [NovelBook_Promotion] {
        var list : [NovelBook_Promotion] = Array()
        for index in 1..<100 {
            let book = NovelBook_Promotion(name: "name_\(index)" , author: "author_\(index)", price: Float(arc4random() % 999))
            list.append(book)
        }
        return list
    }
}

通过分析上述代码，我们发现并没有修改原有的模块代码：接口 Book 与实现类NovelBook 都没有改变，这属于已有的业务，我们保证了历史的纯洁性，降低了风险的扩散！

可能有人会说：增加了 NovelBook_Promotion 类，还是导致了业务层次的修改！高层次的业务层由持久层产生，在业务规划改变的背景下，业务层必须改变来适应新的业务！

注意:开闭原则对扩展开放对修改关闭，并不意味着不做任何修改；低层次模块的变更，必然要有高层模块进行耦合，否则就是一个孤立无意义的代码片段！

1.3、变化的类型

一般而言，项目的变化主要有几种类型：

  • 逻辑变化 :如果仅仅变化一个逻辑，而不涉及其它模块，可以通过修改原有类中的方法来实现！ 提前条件是所有依赖或关联都按相同逻辑来处理！
  • 子模块变化 : 一个模块的变化，必然会引发关联模块的变动；特别是低层次的模块变化引发高层模块的变化！ 因此通过扩展完成变化时，高层的模块修改是必然的！
  • 可见视图变化 ：视图是与用户沟通的第一窗口，它的变化时频繁的、不固定的；一个简单的变化，背后可能牵扯到多张数据表的修改、业务逻辑的重组等！一个健壮的设计，要有足够的灵活来应对多变的视图！

2、为什么要使用开闭原则

为什么要使用开闭原则？开闭原则是最基础的原则：其它五大原则是指导设计的工具与方法，而开闭原则是指导纲领、是精神领袖！

2.1、开闭原则对测试的影响

不要轻易去否定前辈的代码，所有已经投产的代码都有它的历史意义，经过了高压、异常、错误等千锤百炼的测试过程，保证了逻辑正确性！

因此当有新的变化来临时，我们需要考虑：原有的健壮代码是否可以不做修改，而是通过扩展来实现该变化呢？

  • 如果通过修改历史代码来实现变化，那么原有的测试用例需要回测，包括单元测试、功能测试、验收测试等；
  • 想一想引发已有业务的不可预知的风险，就头皮发麻；

一个功能的测试至少包含三个方面：正常的业务逻辑测试，边界条件的测试，异常测试。有些重要模块甚至几十种测试用例，想一想在这样的场景下，你还打算偷懒，对历史代码做微小的改变，来实现变化嘛？

通过扩展来实现变化，避免历史代码的污染！可以仅仅针对扩展来实现单元测试，只要保证扩展的测试通过即可！

2.2、开闭原则可以提高复用性

在面向对象的设计中，所有的逻辑都是从原子逻辑组合而来的，而不是在一个类中独立实现一个业务逻辑！只有这样代码才可以复用，粒度越小被复用的可能性就越大。

为什么要复用？减少代码量，避免相同的逻辑分散在多个角落里，避免日后的维护人员为了修改一个微小的缺陷或者增加一个新功能，而在整个项目里到处查找相关代码！

如何提高复用率呢？缩小逻辑粒度，直到一个逻辑不可再拆分为止！

2.3、开闭原则可以提高可维护性

一个项目上线后，维护人员的工作不仅是对数据进行维护，还可能对程序进行扩展！ 维护人员最乐意做的事情是扩展一个类，而不是修改一个类！ 不管原有的代码写的多么优秀或者糟糕，让维护人员读懂原有代码，然后再修改，都是一件痛苦的事情，是对维护人员的精神摧残与肉体折磨（掉头发）！

2.4、面向对象开发的要求

万物皆对象，我们需要把所有事物都抽象成对象，然后针对对象进行操作！ 但万物每时每刻都在运动、都在变化，有变化就要有策略去应对变化！ 这就要求开发人员在项目设计之初考虑到所有的可能变化因素，为变化预留接口！

3、如何使用开闭原则

开闭原则是一个非常虚的原则，前面五个原则是对开闭原则的具体解释，但是开闭原则并不局限于这么多，它虚的没有边界！就像“好好学习、天天向上”的口号一样，告诉我们要好好学习，但是学什么、怎么学并没有告诉我们，需要我们自己去体会、揣摩。

如何将开闭原则应用于实际的项目中呢？

3.1、抽象约束

  • 通过接口或者抽象类约束扩展，对扩展进行边界限定，不允许出现在接口或者抽象类中不存在的 public 方法；
  • 参数类型、引用对象尽量使用接口或者抽象类，而不是实现类；
  • 抽象层尽量保持稳定，一旦确定即不允许修改。

上文的图书馆系统，只提供的小说书籍太单一，于是后期增加了计算机书籍；它不仅包含书名、作者、价格等信息，还包含一个独特的属性：面向领域（编程领域、数据库领域等）

新增一个计算机书籍的接口，并根据该接口，实现一个计算机书籍的实现类

/// 计算机书籍的接口：继承 书籍接口
protocol IComputerBook : IBaseBook {
    func getScope() -> String
}

/// 计算机书籍：继承基本书籍，有扩展了 计算机书籍接口
class ComputerBook: BaseBook, IComputerBook  {
    private let scope : String

    init(name: String, author: String, price: Float, scope: String) {
        self.scope = scope
        super.init(name: name, author: author, price: price)
    }

    /// 实现新增加的接口方法
    func getScope() -> String {
        return self.scope
    }
}

3.2、元数据控制模块行为

编程是一个枯燥繁重的工作，如何减轻我们的压力呢？答案是尽量使用元数据来控制程序行为，减少重复开发！

什么是元数据？用来描述环境和数据的数据。通俗的讲就是配置参数，参数从文件中获取，也可以从数据库获取。

举个简答例子，登录方法中提供了一些逻辑：先检查 IP 地址是否可以访问，然后再决定是否需要到数据库验证密码！这就是元数据控制模块行为的例子，其中达到极致的就是控制反转 Inversion of Control！

3.3、制定项目章程

一个大的项目团队，建立明确的项目章程是非常重要的；因为项目章程指定了所有人员都必须遵守的约定，对项目而言约定优于配置！

如一个项目中的 Bean 配置文件非常多，管理非常麻烦，如果需要扩展，就需要增加子类并修改 SpringContext 文件。然而如果在项目中有这样一个约定：所有的 Bean 自动注入，使用 Annotation 进行装配，进行扩展时甚至只用写一个子类，然后由持久层生成对象，其它的都不需要修改，这就需要项目内约束，每个项目成员都必须遵守。该方法需要团队成员的自觉性、需要长时间的磨合，一旦团队成员都熟悉这样的规则，比通过接口或者抽象类进行约束效率高，而且扩展性一点也没有减少！

3.4、封装变化

对变化的封装有两层含义：

  • 将相同的变化封装在一个接口或者抽象类中；
  • 将不同的变化封装到不同的接口或者抽象类中，不应该有不同的变化出现在同一个接口或者抽象类中。

封装变化，也就是受保护的变化，找出预计有变化或不稳定的点，我们为这些点创建稳定的接口；准确的讲是封装可能发生的变化，一旦预测到变化，就可以进行封装，后文的 23 种设计模式从不同的角度对变化进行封装！

4、使用开闭原则的注意事项

4.1、开闭原则也只是一个原则

开闭原则仅仅是一个口号，拥抱变化的方法非常多，不必局限于这六大设计原则，但是遵循六大设计原则可以应对大多数变化。

因此在项目中，尽量采取这六大设计原则，适当的时候可以进行扩充，例如通过类文件替换的方式解决系统的一些缺陷！但前提条件是类做到高内聚、低耦合，否则类文件的替换会引起不可预知的故障！

4.2、项目规章非常重要

项目规章是一个团队所有成员共同的知识结晶，是所有成员必须遵守的约定！优秀的项目章程能够给项目带来非常多的好处，如提高开发效率、降低缺陷率、提高团队士气、提高技术成员水平等等！

4.3、预知变化

在实践过程中，一旦发现有发生变化的可能，则需要考虑现有的架构是否可以轻松的实现这一变化。结构师设计一套系统不仅要符合现有需求，还要适应可能发生的变化，这才是一个优良的架构。

开闭原则是一个终极目标，或许无法百分百的达到，但朝这个方向努力，可以显著改善一个系统的结构，做到“拥抱变化”！
